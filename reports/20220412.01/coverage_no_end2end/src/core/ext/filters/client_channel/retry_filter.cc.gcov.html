      46             : // Retry filter
      47             : //
      48             : 
      49             : // This filter is intended to be used in the DynamicFilter stack in the
      50             : // client channel, which is situated between the name resolver and the
      51             : // LB policy.  Normally, the last filter in the DynamicFilter stack is
      52             : // the DynamicTerminationFilter (see client_channel.cc), which creates a
      53             : // LoadBalancedCall and delegates to it.  However, when retries are
      54             : // enabled, this filter is used instead of the DynamicTerminationFilter.
      55             : //
      56             : // In order to support retries, we act as a proxy for stream op batches.
      57             : // When we get a batch from the surface, we add it to our list of pending
      58             : // batches, and we then use those batches to construct separate "child"
      59             : // batches to be started on an LB call.  When the child batches return, we
      60             : // then decide which pending batches have been completed and schedule their
      61             : // callbacks accordingly.  If a call attempt fails and we want to retry it,
      62             : // we create a new LB call and start again, constructing new "child" batches
      63             : // for the new LB call.
      64             : //
      65             : // Note that retries are committed when receiving data from the server
      66             : // (except for Trailers-Only responses).  However, there may be many
      67             : // send ops started before receiving any data, so we may have already
      68             : // completed some number of send ops (and returned the completions up to
      69             : // the surface) by the time we realize that we need to retry.  To deal
      70             : // with this, we cache data for send ops, so that we can replay them on a
      71             : // different LB call even after we have completed the original batches.
      72             : //
      73             : // The code is structured as follows:
      74             : // - In CallData (in the parent channel), we maintain a list of pending
      75             : //   ops and cached data for send ops.
      76             : // - There is a CallData::CallAttempt object for each retry attempt.
      77             : //   This object contains the LB call for that attempt and state to indicate
      78             : //   which ops from the CallData object have already been sent down to that
      79             : //   LB call.
      80             : // - There is a CallData::CallAttempt::BatchData object for each "child"
      81             : //   batch sent on the LB call.
      82             : //
      83             : // When constructing the "child" batches, we compare the state in the
      84             : // CallAttempt object against the state in the CallData object to see
      85             : // which batches need to be sent on the LB call for a given attempt.
      86             : 
      87             : // TODO(roth): In subsequent PRs:
      88             : // - implement hedging
      89             : 
      90             : // By default, we buffer 256 KiB per RPC for retries.
